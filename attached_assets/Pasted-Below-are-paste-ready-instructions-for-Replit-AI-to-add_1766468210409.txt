4) ARABIC NORMALIZATION RULES (CRITICAL)

Implement function normalizeArabic(input):

* Remove harakat and diacritics:
  َ ً ُ ٌ ِ ٍ ْ ّ ٰ and similar marks
* Remove tatweel: ـ
* Normalize hamza variants to base letters:

  * أ إ آ -> ا
  * ؤ -> و
  * ئ -> ي
* Normalize ta marbuta:

  * ة -> ه
* Normalize alif maqsura:

  * ى -> ي
* Remove non-Arabic letters, punctuation, spaces:
  Keep only Arabic letters in the mapping set.

Return normalized string.

This prevents wrong totals and makes input predictable.

────────────────────────────────────────────
5) ABJAD COMPUTATION ALGORITHM

Function computeAbjad(arabicStr):

1. norm = normalizeArabic(arabicStr)
2. sum = Σ abjadMap[letter]
3. reduced = reduceGG(sum) optional, but do not replace sum
4. breakdown = array of {letter, value}

Return:
{ norm, sum, reduced, breakdown }

UI display rules:

* Default display big number = sum
* Show reduced as smaller chip: Reduced: X
* If Abjad Mode is Reduced, use reduced in comparisons that require 1–33 logic

6) HOW TO COMPARE ABJAD VALUES (SOLVING “MAGNITUDE”)

Abjad values can be large. You need deterministic ways to compare them to 1–9 numerology.

Implement two comparison modes (user selectable):

Mode A: Reduced Comparison (simple, user-friendly)

* Use abjadReduced (1–33) as EC-A compare value.
* Pros: aligns with your existing relationType tables
* Cons: discards magnitude information

Mode B: Modular Comparison (keeps magnitude behavior)

* Compare Abjad values by remainder classes:

  * abjadMod9 = sum % 9; if 0 then 9
  * abjadMod11 = sum % 11; if 0 then 11
* Use mod9 as the primary for compatibility with 1–9 Destiny Code.
* Use mod11 only as an “intensity indicator” badge.

Implementation:

* ecACompare = abjadReduced if Abjad Mode Reduced
* else ecACompare = abjadMod9

Also compute:

* intensityBadge = based on sum size:

  * Low: sum < 200
  * Medium: 200–999
  * High: >= 1000
    Show intensity badge on Abjad card.
